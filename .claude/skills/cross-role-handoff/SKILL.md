# Core Principle

交接不是转发结果，而是转移决策上下文。
没有 Why 的交接，接手方无法判断改动是否正确，也无法高效继续工作。

## 检查流程

BEFORE 把任务交给其他协作者:
1. CHECK: 是否包含完整五件套
   - What: 具体改了什么 / 接手方应该看什么
   - Why: 为什么这样做 / 解决什么问题 / 约束是什么
   - Tradeoff: 放弃了哪些方案，为什么没选
   - Open Questions: 还有哪些不确定点
   - Next Action: 希望接手方下一步做什么
2. BLOCK: 如果缺少 Why，禁止交接
3. BLOCK: 如果只有“我改完了你看看”这种结果描述，没有背景和目标，禁止交接
4. PASS: 五件套完整后，才允许发起 handoff

## 输出格式

### Handoff
- What:
- Why:
- Tradeoff:
- Open Questions:
- Next Action:

## Block 场景

### 反例 1：只有 What，没有 Why
@范德彪 我改了三个文件，你帮我 review。

为什么阻止：
- 接手方不知道改动是为了解决什么问题
- 无法判断实现是否符合原始约束
- 很可能重复调研已经被否掉的方案

### 反例 2：只有结论，没有决策背景
我把锁加上了，现在应该没问题了，你继续。

为什么阻止：
- “应该没问题”不是可交接信息
- 没有说明为什么需要锁
- 没有说明是否考虑过无锁方案、事务方案或队列串行化方案

## 通过场景

### 正例
### Handoff
- What:
  我在 `session-store.ts`、`room-state.ts`、`apply-message.ts` 三个文件里加了 CAS 重试逻辑，避免并发写覆盖。
- Why:
  当前多个协作者会同时写同一个 room state。原实现是 read-modify-write，没有版本校验，会出现后写覆盖前写，导致消息丢失。
- Tradeoff:
  我没有选全局 mutex，因为会把所有房间串行化，吞吐太差；也没有先上数据库事务，因为现在状态层还是内存 + 文件持久化，接入成本更高。
- Open Questions:
  CAS 重试次数现在是 3 次，是否足够；高并发下是否需要退避策略还没验证。
- Next Action:
  请范德彪重点 review：
  1. CAS 重试逻辑是否存在竞态漏洞
  2. 测试是否覆盖冲突重试和失败回滚

## 特殊情况

- 如果交接方自己也不清楚 Why → 应该先问小孙确认需求，再交接
- 紧急 hotfix 也不能跳过五件套，但 Tradeoff 和 Open Questions 可以写"时间紧迫，后续补充"
- 多人交接（如黄仁勋 → 范德彪 → 桂芬）：每段交接都需要独立的五件套
