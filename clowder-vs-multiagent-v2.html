架构设计哲学对比

C
Clowder：分层解耦 + 能力堆叠
每层职责清晰分离（路由/执行/存储），通过 CatRegistry 全局注册中心统一管理 provider 配置。能力以"插件"形式叠加（新 provider → 新 AgentService），但随之而来的是复杂度爆炸。

M
Multi-Agent：扁平直连 + 约定优于配置
MessageService 作为单一编排核心，DispatchOrchestrator 内嵌于其中。Runtime Adapter 直接暴露 spawn 逻辑，无额外注册层。代码流程一眼可见。
⚡优：阅读路径短，3文件能理解全局；⚠️劣：新增 provider 需改核心类

3.1 WebSocket 实现

🐾 Clowder — Socket.IO + Room 隔离
// SocketManager.ts
// Room 结构: thread:{threadId}, user:{userId}
io.on('connection', (socket) => {
  socket.on('join_room', ({ threadId, userId }) => {
    socket.join(`thread:${threadId}`)
    socket.join(`user:${userId}`)  // 多Tab同步
  })

  socket.on('cancel_invocation', ({ threadId, invocationId }) => {
    // 验证发送者必须在该 thread 的 room 里
    const rooms = socket.rooms
    if (!rooms.has(`thread:${threadId}`)) {
      socket.emit('error', { code: 'UNAUTHORIZED' })
      return
    }
    invocationTracker.cancel(invocationId)
  })
})

// 广播 — 只发给该 thread 的订阅者
broadcastAgentMessage(threadId, msg) {
  io.to(`thread:${threadId}`).emit('agent_message', msg)
}
// 永远不用 io.emit() 全局广播
⚡ Multi-Agent — 原生 WS + 全局广播
// ws.ts
const sockets = new Set<SocketLike>()

wss.on('connection', (ws) => {
  sockets.add(ws)
  ws.on('message', async (raw) => {
    const event = JSON.parse(raw.toString())
    // 直接交给 MessageService 处理
    await messageService.handleClientEvent(event, ws)
  })
  ws.on('close', () => sockets.delete(ws))
})

// 广播 — 发给所有连接的 socket
broadcast(event) {
  const payload = JSON.stringify(event)
  for (const socket of sockets) {
    if (socket.readyState === WebSocket.OPEN) {
      socket.send(payload)
    }
  }
}
// 无 Room 概念，全量广播
🐾 Clowder 优势： Room 隔离确保消息只推送给有权限的客户端。user:{userId} room 实现多 Tab 状态同步。取消调用需验证 room 成员资格，防越权操作。
👎 代价：引入 Socket.IO 库（~200KB），比原生 WS 重
⚡ Multi-Agent 优势： 零依赖原生 WS，代码 40 行即实现完整通信层。Fastify 5 的 @fastify/websocket 原生集成无额外封装。
👎 代价：多用户场景下所有人收到所有事件，存在信息泄露风险

async resolve(query: string, opts?: ResolveOpts) {
  // 并行搜索 project + global 两个索引

function analyzeA2AMentions(text: string, cats: CatConfig[]) {
  const stripped = stripFencedCodeBlocks(text)
  // 按 pattern 长度降序排列防止短名吃掉长名
  const patterns = sortByLength(buildPatterns(cats))

  return patterns.flatMap(p => {
    const regex = new RegExp(
      `(?:^|\\n)[\\s${BOUNDARY}]*@(${p.pattern})` +
      `(?=[\\s${CJK_PUNCT}]|$)`, 'gim'
    )
    // 过滤自提及、限制最多 2 个目标
    return matchAll(regex, stripped)
      .filter(m => m.catId !== currentCatId)
      .slice(0, MAX_A2A_MENTION_TARGETS) // = 2
  })
}

⚡ Multi-Agent — DispatchOrchestrator 内部调度

// dispatch.ts
class DispatchOrchestrator {
  private readonly MAX_HOPS = 10
  // 每个 root 消息的 hop 计数
  private rootHopCounts = new Map<string, number>()
  // 每个 root 已触发的 provider 集合 (防重复)
  private rootTriggeredProviders = new Map<string, Set<string>>()
  // per-group 串行队列
  private queues = new Map<string, QueuedDispatch[]>()

  enqueuePublicMentions(
    sessionGroupId: string,
    messageRootId: string,
    mentions: MentionMatch[],  // 由 MentionRouter 解析
    sourceProvider: string
  ) {
    const hopCount = this.rootHopCounts.get(messageRootId) ?? 0
    if (hopCount >= this.MAX_HOPS) {
      // 达到上限，记录 blocked dispatch
      this.blockedDispatches.push({...})
      return
    }

    for (const mention of mentions) {
      // 去重：同一根消息链不触发同一 provider 两次
      const triggered = this.rootTriggeredProviders
        .get(messageRootId) ?? new Set()
      if (triggered.has(mention.provider)) continue
      if (mention.provider === sourceProvider) continue

      this.queues.get(sessionGroupId)?.push({
        sessionGroupId, messageRootId,
        targetProvider: mention.provider,
        prompt: '...',  // 原始消息内容
        hopCount: hopCount + 1
      })
    }
  }

  // 串行取出下一个待处理任务
  takeNextQueuedDispatch(sessionGroupId: string) {
    return this.queues.get(sessionGroupId)?.shift()
  }
}

class InvocationQueue {
  async enqueue(invocation: PendingInvocation) {
    this.queue.push(invocation)
    this.broadcastQueueState()
    if (!this.isRunning) await this.processNext()
  }

  async processNext() {
    const next = this.queue.shift()
    if (!next) return
    this.isRunning = true
    try {
      await this.executor(next)
    } finally {
      this.isRunning = false
      await this.processNext()  // 递归处理下一个
    }
  }
}

Zustand 前端状态 — 精简状态管理，chat-store/thread-store/settings-store 职责清晰，无 Redux 样板代码。

15.2 Clowder 的双层权限架构

Layer 1: CLI 层 (bypassPermissions) → Cat CLI 进程内部的文件操作自动执行 → 无弹框、无暂停 → 假设: 外层 Callback 授权层已经把关 Layer 2: Callback 授权层 (AuthorizationManager) → Cat 通过 MCP 调用 request-permission → 触发 POST /api/callbacks/request-permission 流程: ① 查缓存规则 (ruleStore.match) → 有匹配规则 → 直接返回 allow/deny ② 无规则 → 创建 PendingRequest → WebSocket 推送到前端 → 前端弹出授权对话框 ③ 人工决策 (120s 超时) → POST /api/authorization/respond → scope: 'once' | 'thread' | 'global' ④ 如 scope ≠ 'once' → 写入持久规则 → 下次同类操作自动通过 PendingRequestRecord 结构 { invocationId, catId, action, reason, status: 'waiting'|'granted'|'denied' } AuthorizationRule 结构 { catId, action, scope, decision, threadId?, createdAt }

授权粒度范围 (RespondScope)

once — 本次仅此一次，下次同样操作还需确认。用于首次尝试未知操作。
thread — 本 thread 内永久允许，其他 thread 仍需确认。用于当前任务的持续操作。
global — 全局永久允许。用于完全信任的常规操作（如 git commit）。

16.3 根本差距：设计哲学不同

🐾 Clowder：平台思维
把 AI 编排当成一个工程系统来建设。MCP 工具集、授权层、工作流状态机、记忆引擎都是"基础设施"，Agent 在这套基础设施上运行，能力由平台决定而非由 CLI 决定。

结果：Agent 能做的事情很多，但平台本身的复杂度极高，部署、维护、理解都有门槛。
⚡ Multi-Agent：工具思维
把 AI 编排当成一个CLI 协调器来建设。不提供额外基础设施，只负责把 CLI 工具拉起来、传递消息、处理 @mention。Agent 的能力边界 = CLI 工具本身的边界。

结果：简单易部署，但 Agent 的能力天花板是 CLI 版本和配置，很多"高级"操作根本没有触发路径。

🚧 十八、权限机制深析 — Clowder 真的全放行吗？Multi-Agent 为何静默终止？

18.1 Clowder 的 bypassPermissions 不是"完全放行"

误解澄清：bypassPermissions 只是关闭了 Claude CLI 自身的交互确认弹框，并不是放弃所有权限控制。Clowder 在 CLI 之外构建了一套更精细的授权层来补位。
理解 bypassPermissions 的正确方式： 默认 Claude CLI 行为 (acceptEdits): Agent 想写文件 → CLI 弹框 "是否允许写入 xxx.ts?" → 用户点击 ↓ 在服务端自动化中，没有人在点击，请求会永远挂起 bypassPermissions 的含义： Agent 想写文件 → CLI 直接执行，不弹框 ↓ 但"什么文件能写"由上层 Callback 授权层控制 真正的权限控制在 Callback 授权层： Agent 调用 MCP tool (如 execute命令、写关键文件) ↓ POST /api/callbacks/request-permission { invocationId, catId, action, reason } ↓ AuthorizationManager.check(action, catId, threadId) ├─ 匹配到持久规则 (allow/deny) → 立即返回 └─ 无规则 → WebSocket 推送到前端弹框 (120s超时) 用户选择: allow once / allow thread / allow global ↓ 写入 AuthorizationRule 到 Redis 下次同类操作自动放行 所以 bypassPermissions = 把 CLI 层确认挪到 API 层 不是"不确认"，而是"用更好的方式确认"

18.3 静默终止的根本原因 — 完整解析

🔴 场景A：Claude 需要确认但前端无弹框

Claude CLI 执行中，遇到需要确认的操作: 例如: 写入敏感文件、执行 shell 命令 Claude CLI 在 STDERR 输出: "Do you want to proceed? (y/n)" 或: "请确认是否执行此操作？" ↓ message-service.ts extractPromptFromActivityChunk() 检测到 STDERR 中的确认模式 ↓ 把确认文本存入 invocation 的 prompt 字段 广播 WS 事件: { type: 'dispatch.blocked', attempts } ↓ 前端收到 dispatch.blocked 事件 但前端 UI 只有: - 消息气泡 - "停止" 按钮 没有"授权弹框"组件！ ↓ 用户看不到任何提示 CLI 等待 stdin 输入，但 stdin 是 pipe，无人输入 ↓ 30秒无活动 heartbeat 检测 5分钟超时 → SIGTERM → SIGKILL Agent 静默死亡

🔴 场景B：Codex -a never 静默拒绝

Codex 执行中，遇到需要确认的工具调用: 例如: 执行某个 bash 命令 '-a never' 的实际含义: 不是"从不询问"，而是 "遇到需要确认的操作 → 自动拒绝执行" ↓ Codex 在输出中可能只有: "I was unable to execute the command because it requires approval." ↓ 这段文本混在 assistant_delta 流里 用户只看到一串解释性文字 看不出这是权限失败 ↓ 任务实际上没有完成 但前端 UI 显示为"正常结束" 用户以为成功了 // 更糟糕的情况： // Codex 认为自己已经完成（输出了解释） // 触发 @reviewer 的 A2A dispatch // Reviewer 拿到一个"假完成"的代码去 review // 整个链条建立在错误假设上

🔴 场景C：Claude 需要确认但 -a 未设置（最常见的静默失败）

🗂️ 十九、A2A 秩序控制 — 如何防止 Agent 之间的混乱

核心问题：多 Agent 相互 @mention，怎么保证不无限循环、不重复触发、不乱序、不相互打断？

// SessionMutex.ts — invoke-single-cat.ts line 59 引用单例
class SessionMutex {
  private held  = new Map<string, () => void>()
  private waiters = new Map<string, Waiter[]>()

  async acquire(sessionId: string, signal?: AbortSignal) {
    if (this.held.has(sessionId)) {
      // 已被持有 → 排队等待（FIFO）
      return new Promise((resolve, reject) => {
        this.waiters.get(sessionId)!.push({ resolve, reject })
        signal?.addEventListener('abort', () => reject(signal.reason))
      })
    }
    // 快路径：直接获取锁
    return this.lock(sessionId)
  }
}

19.2 并行 A2A 的特殊设计：routeParallel 永不链式触发

// route-parallel.ts (line 10 注释)
// "routeParallel never chains" — 并行模式中 A2A 被完全禁用

// 设计原因：
// 并行执行 A→B, A→C 同时进行
// 如果 B 触发 A2A @D，C 也触发 A2A @D → D 被触发两次
// 如果 B 触发 @C，C 正在执行中 → 状态不一致
// 并行中任何 A2A 链式都会导致不可预测的级联

// 实现：
async function* routeParallel(cats, ...) {
  // 所有 cat 并发执行，共享 teammates 上下文
  const tasks = cats.map(cat => invokeSingleCat(cat, {
    mode: 'parallel',
    teammates: cats.filter(c => c !== cat),
    // 注意：没有传入 a2aEnabled 或 worklistRegistry
    // → 并行模式中 parseA2AMentions 的结果被忽略
  }))
  yield* mergeAsyncIterables(tasks)
}

// 串行 (routeSerial) vs 并行 (routeParallel)
// 串行: 支持 A2A 链式延伸，每个 cat 看到前一个的回复
// 并行: 禁止 A2A，所有 cat 独立回复，互不触发
设计哲学：并行适合"集思广益"（ideate intent），每个 Cat 独立思考不互相影响；串行适合"流水线执行"（execute intent），前一个的输出是下一个的输入，A2A 在此有意义。

19.3 Multi-Agent — 2 层防护及其缺陷

// dispatch.ts — Multi-Agent 的全部 A2A 防护
class DispatchOrchestrator {
  private readonly MAX_HOPS = 10
  private rootHopCounts = new Map<string, number>()
  // 每条消息根的 hop 计数

  private rootTriggeredProviders = new Map<string, Set<string>>()
  // 每条消息根已触发过的 provider 集合

  enqueuePublicMentions(sessionGroupId, messageRootId, mentions, sourceProvider) {
    const hopCount = this.rootHopCounts.get(messageRootId) ?? 0

    // 防护1：hop 上限
    if (hopCount >= this.MAX_HOPS) {
      this.blockedDispatches.push({ reason: 'hop_limit', ... })
      return
    }

    for (const mention of mentions) {
      const triggered = this.rootTriggeredProviders.get(messageRootId) ?? new Set()

      // 防护2：同根消息链不重复触发同一 provider
      if (triggered.has(mention.provider)) continue
      if (mention.provider === sourceProvider) continue

      triggered.add(mention.provider)
      this.queues.get(sessionGroupId)?.push({ ... })
    }
  }
}

19.4 Multi-Agent A2A 的具体缺陷逐条分析

❌ 缺陷1：rootTriggeredProviders 去重太激进

// 问题：同一根消息链，每个 provider 只能被触发一次
// 场景：
// 用户: "实现登录功能"
// Claude: "@Codex 写代码"      → Codex 入队，触发记录
// Codex: 写完 "@Claude review"  → Claude 触发记录
// Claude review 完: "@Codex 有bug" → ❌ Codex 已在记录里，跳过！

// Codex 永远没机会修复它自己发现的 bug
// 工作流在第二轮就断掉了

// Clowder 的做法：
// 使用 Pending-tail 去重（只查待执行列表）
// 已执行完的 cat 可以重新入队
// A→B→A 是合法的修复迭代
根去重策略在"同一问题需要多轮迭代"的场景下会提前断链，导致工作流无法完成。

❌ 缺陷2：hop count 是根级别的，非链级别

// rootHopCounts 按 messageRootId 计数
// messageRootId = 用户原始消息的 id

// 问题：如果用户消息触发了很长的协作链
// 每一跳都消耗这个根的配额
// 用完 10 跳后，即使只是简单的确认消息也被阻断

// 更严重的：不同 session group 的计数互不干扰
// 但同一 group 内的所有 A2A 共享一个根的计数
// 如果链条中某个 Agent 发了多条消息，每条都算一跳

// Clowder 的做法：
// a2aCount 按 worklist 全局计数，但不与根绑定
// 同一个 agent 发多条消息只算一次扩展
// 更精确地反映"实际 A2A 跳转次数"
hop 计数颗粒度粗，容易误伤正常的多条消息交流，把合法协作当成无限循环截断。

❌ 缺陷3：完全没有用户优先机制

// 但 B 和 C 还在排队执行
// 用户无法插入新消息（isRunning 标志阻止）
// 只能等待整条链结束（最长 15 分钟）

❌ 缺陷4：无 SessionMutex，同 CLI 会话可能并发

Multi-Agent 把 A2A 当成"简单的消息转发"来设计——够用但脆弱，rootTriggeredProviders 去重策略把"防无限循环"和"允许合理迭代"混为一谈，导致标准的开发→review→修复工作流都跑不通。

🧩 二十、精确分工场景深析 — "A分析 → B改代码 → A+C一起review"

具体场景：用户说 "帮我看看这个 bug"
期望结果：A（架构师）分析问题 → @B（开发者）修代码 → B修完后 @A 和 @C 一起 review
核心挑战：B修代码时，A和C不能也去改代码；A和C review时，B不能再发起新修改；每个人要能看到前面人做了什么

20.2 "只让 B 改代码" 的真正保障机制 — 不是硬锁，是软约束叠加

1
worklist 顺序隔离
B 执行时 A 和 C 还没有执行，物理上不可能同时改代码。串行 worklist 保证同一时刻只有一个 Cat 在运行。
2
directMessageFrom 语义约束
B 被 A @mention，prompt 里说"A 让我来修代码"。B 的上下文清晰：任务是按 A 的分析修代码，不是做其他事。
3
品种触发词角色约束
dare 的触发词："严格按指令执行，不添加无关改动"
codex 的触发词："不做功能开发，只做 review"
系统 prompt 级别的行为约束。
4
previousResponses 上下文感知
A 第二次执行时已知道 B 修了什么。C 执行时看到 A 和 B 都做了什么。没有信息盲区，不会重复工作。
⚠️ 重要澄清：Clowder 没有"硬锁"

Clowder 不会在代码层面阻止 A 去修改文件。如果 A 真的想改代码，它有 MCP 文件工具。

真正的约束是：
① 角色 prompt 让 A 知道自己是架构师，不是编码者
② 上下文让 A 知道 B 已经修好了
③ directMessageFrom 让 A 知道现在是被叫来 review 的

这是"认知约束"而非"执行锁"。
如果 A 的 system prompt 足够清晰，A 不会想去改代码。
如果 A 真的改了代码（AI 幻觉），Callback 授权层可以拦截。

需要改造的 6 个部分（按难度排序）

#改造内容对应 Clowder 实现难度说明
1 per-provider system prompt 差异化 WORKFLOW_TRIGGERS per breed 低 直接修改 agent-prompts.ts，给 Claude/Codex/Gemini 写不同的角色描述和工作流触发词

// ═══ 改造1：dispatch.ts — 改用 pending-tail 去重 ═══
// 现在：
private rootTriggeredProviders = new Map<string, Set<string>>()
// if (triggered.has(mention.provider)) continue  ← 删掉这行

// 改为：
private pendingQueues = new Map<string, Set<string>>()
// 只检查"当前已排队但未执行"的 provider

enqueuePublicMentions(...) {
  const pending = this.pendingQueues.get(sessionGroupId) ?? new Set()
  for (const mention of mentions) {
    if (mention.provider === sourceProvider) continue  // 自提及过滤
    if (pending.has(mention.provider)) continue        // 已在待执行队列里
    pending.add(mention.provider)
    this.queues.get(sessionGroupId)?.push(...)
  }
}

takeNextQueuedDispatch(...) {
  const next = queue.shift()
  // 执行完后从 pending 中移除
  this.pendingQueues.get(next.sessionGroupId)?.delete(next.targetProvider)
  return next
}
// 效果：A→B→A 第二轮可以执行（A 已经出了 pending）

// ═══ 改造2：dispatch.ts — 在 content 里携带完整链上下文 ═══
// 现在：

  `You were mentioned by ${sourceAlias}.`,
  `Latest public message: ${latestMessage}`,

// 改为：

].join('\n')

🔏 二十一、权限控制深度剖析 — Claude Code / Codex / Gemini CLI 各自是怎么做的

🎯 核心问题：为什么 Multi-Agent 没有权限控制

三个 CLI 工具各自都有权限控制机制，但 Multi-Agent 把它们全部关掉了，而且没有在外层建立任何替代机制。要理解为什么，先要理解每个 CLI 的权限设计。

① Claude Code 的四种权限模式

Claude Code CLI 通过 --permission-mode 参数控制工具使用时是否暂停等待人工确认：

权限模式 文件读 文件写/编辑 Bash命令执行 交互方式
default ✓ 自动 ⚠ 弹确认框 ⚠ 弹确认框 CLI 交互式 stdin 等待用户输入
acceptEdits ✓ 自动 ✓ 自动 ⚠ 弹确认框 只有 bash/命令需要确认
bypassPermissions ✓ 自动 ✓ 自动 ✓ 自动 全部跳过，CLI 从不暂停
plan ✓ 自动 ✗ 禁止 ✗ 禁止 只读模式，只能规划不能执行
⚠️ bypassPermissions 的真正含义： Claude CLI 本身完全不会暂停等待人工确认。文件写入、bash 命令、一切工具调用都立即执行。 这看起来很危险——但 Clowder 用这个模式的理由是：它在 CLI 外层建立了自己的权限系统。

Codex CLI 用 -a / --approval-mode 和 -s / --sandbox 两个参数控制权限：

-a / --approval-mode 取值

值含义
suggest (默认)每次执行前都询问用户：是否继续？
auto-edit文件编辑自动批准，bash命令仍询问

-s / --sandbox 取值

值含义
none无沙箱，可访问整个文件系统
read-only只读沙箱，不能写任何文件
workspace-write只能写当前工作目录内的文件
full完整沙箱（需要 Docker/VM 支持）

🔎 问题 2：C 是否基于"A 改之前"的代码来 review

串行执行保证了时序正确性 — 这个担心是多余的

场景：A 修改 → @B @C review → B review 后 → @A 继续改 → C 此时 review 的是哪个版本？ Worklist 演化： ┌──────────────────────────────────────────────────────────────┐ │ 初始： [A] 用户把任务给 A │ │ ↓ A 执行完，说 "@B @C review" │ │ 扩展1： [A✓, B, C] a2aFrom: {B→A, C→A} │ │ ↓ B 执行完，说 "@A @C 请修复以下3个问题" │ │ 扩展2： [A✓, B✓, C, A] a2aFrom: {C更新为→B, A→B} │ │ ↓ C 接下来执行（第3步） │ │ C执行： [A✓, B✓, C✓, A] ← C 是第3个执行的 │ │ ↓ A 最后执行（第4步，基于 B 和 C 的 review 修改） │ │ A执行： [A✓, B✓, C✓, A✓] ← A 重新修改 │ └──────────────────────────────────────────────────────────────┘ 关键：C 在 A 重新修改"之前"执行 ✓ A 重新修改时已经拥有 B 的 review + C 的 review ✓ 这正是正常的代码 review 流程！
正常 code review 就是这个顺序：
A 写代码 → B review → C review（基于 A 的原始代码）→ A 根据 B+C 两人的 review 统一修改
C 不需要等 A 修改完再 review——C 是 reviewer，C 的工作就是 review A 的当前代码，给 A 提意见。

    // C 已经在队列里了
    // 就算 B 再@C，也不会重复添加
    // (但会更新 a2aFrom)
    continue;
  }
  worklist.push(nextCat);  // 只有不在队列里才加
}
// A→B→A 的 ping-pong 允许（A已执行，不在pending里）
// A→B→C→B（B还没执行，不重复加）
// ── 防护4：parseA2AMentions 每次最多2个目标 ──
// a2a-mentions.ts
// strip code fences → longest-match-first
// → line-start only → token boundary
// → self-filter → max 2 targets per turn

// 一个 Agent 每轮最多只能@2个人
// 避免"我@所有人"式的爆炸式扩展
// 即使深度15，最差情况也是 2^15
// 但 pending-tail 去重大幅限制实际数量

四重防护合力：深度15跳（硬停）+ 用户消息抢占（软停）+ pending去重（防快速振荡）+ 每轮最多2目标（防爆炸扩展）

⚠️ 边界情况：如果 C 是通用 breed，真的尝试"既 review 又改"

class SerialExecutionQueue {
  private queue: DispatchTask[] = []
  private running = false

  async enqueue(task: DispatchTask) {
    this.queue.push(task)
    if (!this.running) await this.drain()
  }

  private async drain() {
    this.running = true
    while (this.queue.length > 0) {
      const task = this.queue.shift()!
      await this.execute(task)
      // 执行完后，从 pendingProviders 移除
    }
    this.running = false
  }
}
