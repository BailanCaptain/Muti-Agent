# Bug 诊断胶囊模板

Phase 1 入口：复制下方模板，逐栏填写，驱动调查过程。

---

## 诊断胶囊

| # | 栏位 | 内容 |
|---|------|------|
| 1 | **现象** | <!-- 看到了什么？错误信息原文，粘贴完整 stack trace --> |
| 2 | **证据** | <!-- 日志片段 / 截图 / 复现步骤（精确到命令） --> |
| 3 | **假设** | <!-- 初步猜测根因是什么，列 1-3 个候选 --> |
| 4 | **诊断策略** | <!-- 接下来验证什么、怎么验证（具体命令/断点/日志） --> |
| 5 | **超时策略** | <!-- 如果 N 分钟没进展：换假设 / 加诊断桩 / 升级 --> |
| 6 | **预警策略** | <!-- 什么情况下升级给小孙（数据风险 / 用户可见 / 阻塞其他人） --> |
| 7 | **用户可见修正** | <!-- （可选）需要先告知小孙的影响，如：服务暂时降级 --> |
| 8 | **验收** | <!-- 怎么确认修好了（测试名 / 命令 / 预期输出） --> |

---

## 使用说明

- **入口**：遇到 bug → 先填胶囊再动手，不要跳过
- **迭代**：每轮假设验证后更新栏 3（假设）和栏 4（策略）
- **出口**：根因确认 → 进入 Phase 4 写失败测试 → 修复
- **存档**：修复完成后，从胶囊提炼 Bug Report 五件套

## 填写示例

| # | 栏位 | 内容 |
|---|------|------|
| 1 | **现象** | `TypeError: Cannot read properties of undefined (reading 'id')` at `message-service.ts:142` |
| 2 | **证据** | 仅在 A2A 消息触发时出现，普通用户消息正常。日志显示 `dispatch.targetAgent` 为 undefined |
| 3 | **假设** | H1: flushDispatchQueue 没正确解析 @mention → targetAgent 为空 |
| 4 | **诊断策略** | 在 `flushDispatchQueue` 入口加 `console.log(dispatch)` 看完整对象 |
| 5 | **超时策略** | 15 分钟没进展 → 对照正常消息的 dispatch 对象做 diff |
| 6 | **预警策略** | 如果影响所有 A2A 消息 → 立即告知小孙 |
| 7 | **用户可见修正** | 无，仅影响 agent 间通信 |
| 8 | **验收** | `pnpm test` 通过 + 手工发一条 A2A 消息确认不报错 |
