# Core Principle

Review 的目的不是让作者开心，而是阻止错误进入主干。
没有问题分级的 review，等于没有执行标准。

## 检查流程

WHEN 做 review:
1. CHECK: 是否真正检查了实现，而不是只给总体评价
2. CHECK: 是否输出了具体问题，而不是泛泛而谈
3. CHECK: 每个问题是否按 P1 / P2 / P3 分级
4. CHECK: 是否明确说明哪些问题会阻断放行
5. BLOCK: 禁止输出无信息量评价
   - “Looks good”
   - “代码很清晰”
   - “整体 OK”
   - “没什么大问题”
6. PASS: 只有在问题、级别、影响、建议都明确时，review 才算完成

## 分级标准

## P1/P2/P3 分级标准

| 级别 | 定义                                       | 处理方式             |
| ---- | ------------------------------------------ | -------------------- |
| P1   | 阻断级：会导致功能错误、数据丢失或安全问题 | 必须立即修复         |
| P2   | 重要：代码质量问题、测试覆盖不足、性能隐患 | 必须修完才能放行     |
| P3   | 建议：风格、命名、可以更好但不影响功能     | 登记 BACKLOG，可商量 |

## 输出格式

### Review Findings

#### P1
- [文件/位置] 问题
  - 为什么是问题:
  - 影响:
  - 建议修复:

#### P2
- ...

#### P3
- ...

### Review Verdict
- NOT APPROVED
或
- APPROVED

## 通过场景

### 正例
### Review Findings

#### P1
- `conversation-store.ts:87`
  - 问题: 当前 `save()` 是 read-modify-write，没有版本校验。多个协作者并发写入时，后写会覆盖前写。
  - 为什么是问题: 这会直接造成消息丢失，属于功能正确性错误。
  - 影响: 多协作者聊天场景下会出现历史缺失。
  - 建议修复: 加入版本号 + CAS，或者把写路径改成单线程串行消费。

#### P2
- `agent-router.test.ts`
  - 问题: 没有覆盖“双协作者同时被 @”的顺序和幂等测试。
  - 为什么是问题: 当前逻辑依赖事件时序，没有测试很容易回归。
  - 影响: 后续重构时无法保证行为稳定。
  - 建议修复: 至少补充并发入场与重复事件测试。

#### P3
- `parseMentionTargets`
  - 问题: 函数名更像纯解析，但实际还做了过滤和规范化。
  - 建议修复: 可考虑改名为 `resolveMentionTargets`。

## 特殊情况

- 如果代码量很大（>500 行），可以分批 review，但每批都要遵守分级规则
- 如果缺少 Why（不知道代码要解决什么问题），先要求补全上下文再 review
- 如果对某个领域不熟（如密码学、分布式一致性），主动声明并建议找专业的人确认
