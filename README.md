# 多模型控制台

这是一个本地网页项目，用来把三个命令行 AI 工具统一放到一个页面里使用：

- Codex
- Claude Code
- Gemini CLI

你可以把它理解成一个“本地 AI 控制台”：

- 上面有一个总输入框
- 可以一条消息同时发给三个模型
- 每个模型下面也有自己的单独输入框
- 每个模型都保留自己的会话线程
- 如果某个模型回答太久，可以单独点“停止”

这份 README 偏“使用说明 + 方案说明”。  
如果你想看更偏底层原理的解释，再看 [ARCHITECTURE.md](C:\Users\-\Desktop\Multi-Agent\ARCHITECTURE.md)。

## 一、这个项目到底解决了什么问题

如果你平时分别用：

- `codex`
- `claude`
- `gemini`

那么以前通常要：

1. 打开多个终端
2. 分别输入问题
3. 分别看输出
4. 自己记住每个工具的上下文和状态

这个项目把这些动作集中到一个网页里，让你可以：

- 在一个页面里同时观察三个模型
- 一条消息同时发给三个模型
- 单独给某一个模型发消息
- 单独停止某一个模型
- 在本地保存每个模型各自的会话历史

## 二、怎么启动

在项目目录下运行：

```bash
node server.js
```

或者：

```bash
npm start
```

然后打开浏览器访问：

```text
http://localhost:3000
```

如果 `3000` 端口被占用了，可以换一个端口。

PowerShell 示例：

```powershell
$env:PORT=3210
node server.js
```

然后访问：

```text
http://localhost:3210
```

## 三、这个项目用了什么技术栈

这个项目故意选了一套很轻、很容易读懂的技术栈。

### 前端

- HTML
- CSS
- 原生 JavaScript

### 后端

- Node 自带的 `http`

## 四、为什么前端这样选型

这一段是你特别要求补细的部分。

### 当前前端选型

现在前端没有用 React、Vue、Svelte 这类框架，而是直接使用：

- HTML 写结构
- CSS 写样式
- 原生 JavaScript 写逻辑

### 为什么这样选

当前项目的目标不是做一个超大型复杂管理后台，而是先把以下几个关键能力做出来：

1. 有一个能用的网页界面
2. 能发请求给后端
3. 能接收流式回复
4. 能实时更新页面
5. 不需要额外前端构建工具

在这个阶段，原生前端的优势非常明显：

- 依赖少
  不需要先安装 React、Vite、Webpack、Babel 之类一堆东西。

- 启动简单
  后端一跑，页面就能直接访问。

- 学习成本低
  对小白来说，直接看一个 `app.js`，比先理解组件、状态管理、构建链条更容易。

- 排查问题简单
  当前页面逻辑集中在 `public/app.js`，出问题时更容易顺着函数直接看。

- 更贴合现在的项目规模
  当前页面虽然比最开始复杂了一些，但还没有复杂到必须上框架。

### 这种前端选型的优点

- 易读
- 易改
- 不需要打包
- 不需要额外前端工程配置
- 对本地单机工具很合适

### 这种前端选型的缺点

- 页面逻辑继续变复杂后，代码组织会越来越吃力
- 没有组件系统，复用性一般
- 状态越来越多时，维护难度会上升
- 手工操作 DOM 的代码会越来越多
- 如果以后要做更复杂的 UI 交互，开发效率可能下降

### 如果以后升级前端，可以怎么改

后续如果页面继续变复杂，比较合理的升级路径是：

1. 升级到 React + Vite
   适合把界面拆成组件，例如：顶部广播区、单个 provider 卡片、消息列表、状态条。

2. 引入更清晰的状态管理
   比如把“线程状态、运行状态、流式事件状态”从散落变量收敛到统一状态容器。

3. 把网络层抽出来
   把当前直接写在 `app.js` 里的 fetch 和 stream 解析逻辑抽成独立模块。

4. 做更明确的 UI 设计系统
   例如按钮、标签、状态色、提示框统一抽象。

## 五、为什么后端这样选型

### 当前后端选型

当前后端使用的是：

- Node.js
- 原生 `http`
- 原生 `child_process`

也就是说，没有上 Express、Fastify、NestJS 这些框架。

### 为什么这样选

因为当前后端核心任务其实只有三类：

1. 接收前端请求
2. 启动和停止本地 CLI 进程
3. 把 CLI 输出流式转发给前端

这些需求用 Node 原生能力就足够完成。

之所以现在不急着上框架，是因为：

- API 数量很少
- 没有复杂中间件体系
- 没有复杂权限系统
- 没有复杂数据库模型
- 最核心的难点其实不是“路由”，而是“进程管理 + 流式返回”

而 Node 原生对这两件事本来就支持得很好。

### 这种后端选型的优点

- 依赖少
- 启动快
- 结构简单
- 更容易看到 HTTP 请求和响应的本质
- 对流式返回的控制更直接
- 对本地进程启动和停止更直接

### 这种后端选型的缺点

- 随着接口增多，原生 `http` 的可维护性会变差
- 没有框架提供的中间件生态
- 路由、参数校验、错误处理都要手写
- 如果以后变成多人使用的服务，扩展会更麻烦

### 如果以后升级后端，可以怎么改

后续如果后端复杂度上来，推荐这样升级：

1. 迁移到 Express 或 Fastify
   好处是路由、参数校验、中间件、错误处理会更规范。

2. 把“进程管理”单独拆成服务层
   当前 `server.js` 既管 API，也管进程生命周期。后续可以拆成：
   - 路由层
   - 运行控制层
   - 存储层

3. 加统一日志系统
   比如为每次请求生成 request id，为每个 provider run 生成 run id。

4. 引入数据库
   当前 JSON 适合本地原型，不适合更大规模数据。

## 六、为什么存储先选 JSON 文件

### 当前方案

现在使用本地 JSON 文件：

- `data/store.json`

### 为什么这样选

因为当前是本地单人使用工具，不是多用户线上系统。

JSON 文件的优点是：

- 不需要安装数据库
- 不需要写迁移脚本
- 不需要先设计复杂表结构
- 直接打开文件就能看到保存了什么

### 这种方案的缺点

- 并发写入能力弱
- 数据量大时性能会越来越差
- 查询能力差
- 不适合多用户同时使用

### 后续推荐改进

最合适的下一步是：

- SQLite

原因是它兼顾了：

- 本地部署简单
- 没有独立数据库服务
- 比 JSON 更可靠
- 能支持结构化查询

## 七、为什么前后端通信先选 HTTP + 流式 NDJSON

### 当前通信方案

项目现在有两种通信方式：

1. 普通 JSON 请求
2. 流式 NDJSON 请求

### 为什么不用纯轮询

如果只用轮询，那么前端必须不断去问后端：

- “现在回复到哪了？”
- “现在结束了吗？”

这样会导致：

- 请求很多
- 实时性差
- 逻辑更绕

### 为什么不用 WebSocket 起步

WebSocket 当然可以做，而且以后也能升级。

但当前场景本质上是：

- 前端发一条消息
- 后端启动一次模型运行
- 后端把这次运行的输出持续返回

这其实非常适合用：

- 一次 HTTP 请求
- 一个持续打开的响应流

所以现在用 NDJSON 流更直接。

### 当前方案的优点

- 实现简单
- 浏览器原生支持
- 不需要额外协议层
- 很适合“一问一答，但回答是流式”的场景

### 当前方案的缺点

- 如果以后要做更多“服务端主动推送”，会显得不够灵活
- 如果以后要做更复杂的双向实时控制，WebSocket 会更自然

### 后续推荐改进

如果未来出现这些需求：

- 多页面实时同步
- 更复杂的服务端主动推送
- 更强的连接状态管理

那么可以升级为：

- SSE
- 或 WebSocket

## 八、当前方案的整体优劣

### 优点

- 依赖少
- 启动快
- 容易理解
- 本地运行友好
- 适合快速验证产品方向
- 并发和停止能力已经具备

### 不足

- 前端结构后续容易变重
- 后端原生 `http` 后续不够优雅
- JSON 存储不是长期方案
- 还没有真正的历史线程管理界面
- 还没有更完整的日志和监控

## 九、后续建议的升级路线

如果继续迭代，我建议按这个顺序升级：

1. 把存储从 JSON 升级到 SQLite
2. 增加历史线程切换能力
3. 增加日志和调试信息
4. 把前端逐步升级为 React + Vite
5. 把后端逐步升级为 Express/Fastify
6. 如果通信更复杂，再升级到 SSE / WebSocket

## 十、项目目录说明

- `server.js`
  后端入口，负责 API、静态文件服务、启动/停止模型进程

- `lib/ai-runtime.js`
  三个 CLI 的统一适配层，负责真正把进程拉起来，并解析输出

- `lib/store.js`
  本地存储层，负责把对话写进 `data/store.json`

- `public/index.html`
  网页结构

- `public/styles.css`
  网页样式

- `public/app.js`
  前端逻辑，负责监听按钮、发请求、读取流式回复、刷新界面
