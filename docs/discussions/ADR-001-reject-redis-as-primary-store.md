# ADR-001: 当前阶段否决 Redis 直替 SQLite 主存储

**状态**: accepted
**日期**: 2026-04-14
**决策者**: 黄仁勋、范德彪、桂芬（三方共识）
**审批**: 小孙（CVO）

## 背景

系统在多 agent 并发场景下体感卡顿。团队讨论了两个方向：
- 方案 A：保留 SQLite，优化查询模式 + 增量快照（2-3 天）
- 方案 B：一次性将 SQLite 替换为 Redis 作为主存储（7-10 天）

## 决策

**否决方案 B（Redis 直替主存储）。采用方案 A（优化现有 SQLite + 增量协议）。**

Redis 保留为未来的**缓存层/事件通道**候选，不作为主存储。

## 理由

### 技术理由
1. 卡顿 Top 3 根因（全量快照重建 40%、前端重渲染 30%、N+1 查询 20%）与存储引擎无关，换 Redis 不解决
2. SQLite 当前量级（messages 1474 条）远未到性能天花板，瓶颈在访问模式
3. agent_events 写放大（44857 条）是被忽视的大瓶颈，换 Redis 只是"把噪音写得更快"

### 工程理由
4. Redis 改动涉及 25-30 个文件，全量数据模型重写，回滚 = 重做
5. Windows 11 环境 Redis 官方不支持
6. 新增运维依赖（Redis 进程管理、持久化配置）
7. 内存开销增加 100MB+

### 数据安全理由
8. 小孙明确要求"不能删除改变原始数据"，Redis 迁移存在数据丢失风险
9. SQLite 嵌入式架构零依赖，数据安全性更高

## 重新评估条件

当且仅当以下条件**同时成立**时重新评估：
1. 查询模式已优化（N+1 消灭、增量快照已实现）
2. 并发 agent > 10，SQLite 写锁成为实测瓶颈
3. 需要跨进程共享状态（多实例部署）
